# Plan De Travail - Points 1 A 4

Date: 2026-03-08  
Scope: Lausanne (MVP operationnel), avec extension progressive.

## Objectif global

Passer d'un moteur "point" a un produit utilisable sur carte avec:

1. calcul d'ensoleillement sur zone (`/api/sunlight/area`)
2. visualisation frontend interactive (carte + couches soleil/ombre)
3. exploitation de lieux (parcs/terrasses) avec fenetres horaires
4. meilleure precision des ombres batiments

## Point 1 - API zone

### Livrables

- Endpoint `POST /api/sunlight/area`
- Entrees:
  - `bbox` (lon/lat)
  - `date`, `timezone`
  - `gridStepMeters`
  - mode `instant` (heure locale) ou `daily`
- Sorties:
  - grille des points evalues
  - statut `isSunny`, `terrainBlocked`, `buildingsBlocked`
  - metadonnees de calcul (points, duree, warnings)

### Critere d'acceptation

- Reponse JSON stable avec validation Zod
- Temps de reponse acceptable sur bbox urbaine (MVP)

## Point 2 - Frontend carte

### Livrables

- Carte interactive centree Lausanne
- Controle de date + heure + resolution grille
- Appel `POST /api/sunlight/area`
- Affichage couche de points soleil/ombre sur la carte

### Critere d'acceptation

- UX utilisable desktop/mobile
- Rafraichissement des resultats sans rechargement de page

## Point 3 - Parcs / terrasses

### Livrables

- Ingestion POI Lausanne (OSM Overpass): parcs + terrasses (cafe/bar/restaurant/pub/biergarten)
- Fichier de donnees versionne sous `data/raw/osm` (ignore git) + manifest
- Endpoint `GET /api/places` pour lister les lieux
- Endpoint `POST /api/places/windows` pour fenetres d'ensoleillement par lieu/date

### Critere d'acceptation

- Chaque lieu retourne des fenetres horaires + indicateurs de blocage

## Point 4 - Precision batiments

### Livrables

- Evolution preprocess batiments:
  - de `bbox` vers `footprint polygonal` (convex hull minimum)
- Evolution moteur ombre:
  - intersection rayon -> polygone 2D extrude (prisme vertical)

### Critere d'acceptation

- Methode exposee dans la reponse API (ex: `dxf-footprint-prism-v1`)
- Reduction des faux positifs de blocage vs modele bbox

## Sequencement implementation

1. Point 1 (`/api/sunlight/area`)
2. Point 2 (carte interactive)
3. Point 3 (POI + windows)
4. Point 4 (empreintes polygonales)

## Risques et mitigations

- Performance sur grandes bbox:
  - limiter points max, pagination ou tuilage
- Donnees POI heterogenes:
  - normaliser schema et tags OSM utiles
- Precision geometrique:
  - versionner les methodes et comparer avant/apres
