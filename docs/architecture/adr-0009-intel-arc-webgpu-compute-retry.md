# ADR-0009 - Fiabilité WebGPU compute sur Intel Arc et réessai encadré

**Date** : 2026-04-13
**Statut** : Accepté
**Contexte** : Le backend `webgpu-compute` visait un calcul d'ombres plus efficace que le raster GPU actuel, mais les essais précédents ont montré des crashs dans le chemin Node `webgpu` / Dawn / D3D12 sur matériel Intel Arc / Arc-class.

## Problème

Nous n'avons pas de source publique qui prouve que notre crash local correspond exactement à un bug Intel Arc documenté.

Les faits locaux sont cependant suffisants pour traiter ce chemin comme expérimental :

- `webgpu-compute` utilise Dawn via le package npm `webgpu`.
- Les commentaires existants du dépôt identifient déjà des risques de segfault Dawn/D3D12 autour de la sortie de process, des workers forkés et de l'ordre entre initialisation Dawn et I/O fichiers.
- `gpu-raster` reste le chemin d'accélération supporté par défaut pour le precompute.
- L'approche "serveur GPU spécial qui redémarre quand il crash" a été rejetée : elle masquait le problème mais l'overhead de redémarrage et de réinitialisation rendait le pipeline trop coûteux.

Ce diagnostic doit rester conservateur : nous avons des symptômes locaux et des bugs voisins documentés, pas une preuve publique d'un bug driver identique.

## Indices externes

Les sources externes montrent que l'écosystème Dawn/WebGPU gère déjà des différences de backends et des contournements matériels, mais elles ne documentent pas notre crash exact.

- `node-webgpu` publie `dawn.node`, expose des options de création Dawn, permet de choisir un backend comme `d3d12` ou `vulkan`, et indique que la durée de vie de `dawn.node` dépend des références retournées par `create()`. Les bugs Dawn/WebGPU doivent être remontés dans le tracker Chromium. Source : https://github.com/dawn-gpu/node-webgpu
- Dawn contient plusieurs toggles de contournement pour des bugs Intel/D3D12/Gen12, notamment sur l'initialisation des textures depth/stencil, les clear values D3D12 et certaines opérations shader/copy. Source : https://dawn.googlesource.com/dawn/+/refs/heads/chromium/7608/src/dawn/native/Toggles.cpp
- ONNX Runtime WebGPU a un ticket ouvert sur des prédictions incorrectes en Electron sur plusieurs GPU Intel, avec un workaround applicatif consistant à exécuter des tests d'initialisation pour désactiver le modèle si les sorties sont invalides. Cela montre qu'un produit commun peut rencontrer des problèmes WebGPU/Intel spécifiques sans que ce soit le même crash que le nôtre. Source : https://github.com/microsoft/onnxruntime/issues/24442
- Intel documente ORT-WebGPU sur Intel Arc via Dawn et note que Dawn peut passer par D3D12 ou Vulkan. Pour l'accélération Intel XMX de l'article, le chemin documenté active Vulkan et désactive D3D12. Source : https://www.intel.com/content/www/us/en/developer/articles/community/boost-ai-inference-performance-with-webgpu.html

## Décision

Nous gardons `gpu-raster` et le fallback CPU comme chemins supportés en production.

Nous n'abandonnons pas définitivement `webgpu-compute`, mais tout nouveau test doit être un réessai encadré :

1. `webgpu-compute` reste expérimental via `MAPPY_BUILDINGS_SHADOW_MODE=webgpu-compute`.
2. Le GPU doit être porté par un sous-processus lancé avec `spawn()`, pas par `fork()` et pas par un redémarrage par tuile.
3. Le sous-processus GPU doit être long-lived : il initialise Dawn une fois, traite beaucoup d'évaluations, puis détruit explicitement le device avant sortie.
4. Le process GPU ne doit pas faire d'I/O fichier après l'initialisation Dawn ; le process parent prépare les données et persiste les artefacts.
5. Le réessai doit comparer explicitement `backend=d3d12` et `backend=vulkan`. Si Vulkan reste bloqué au chargement, `backend=d3d11` peut servir de contrôle expérimental non-D3D12, sans être considéré comme un chemin de production.
6. Le réessai doit vérifier `GPUDevice.destroy()`, le nettoyage des références `create()`, les timeouts et le chemin `device.lost` quand il est observable.

Le prototype local ajoute les flags internes d'expérimentation `MAPPY_WEBGPU_BACKEND=d3d12|d3d11|vulkan|auto` et `MAPPY_WEBGPU_DLLDIR`. Ils ne constituent pas une API publique.

## Observations expérimentales du 2026-04-13

Un premier probe minimal a permis de séparer plusieurs problèmes :

- Le cycle minimal Dawn/D3D12 fonctionne : création adapter/device, compute trivial répété, readback, `GPUDevice.destroy()` et `device.lost` avec `reason=destroyed`.
- Le runtime Vulkan système est présent sur la machine testée : `vulkaninfo` voit l'Intel Arc 140V, `vulkan-1.dll` est présent dans `C:\Windows\System32` et un chargement système direct de la DLL réussit.
- Le backend Vulkan n'est pas encore disponible via `node-webgpu` : Dawn échoue à charger `vulkan-1.dll` (`Windows Error: 87`) avant de retourner un adapter. L'échec observé est donc spécifique au chemin Dawn/node-webgpu, pas à une absence générale de Vulkan sur la machine.
- Le probe `benchmark:webgpu:vulkan-loader` a testé plusieurs variantes sans succès : environnement standard, `C:\Windows\System32` en tête de `PATH`, `dlldir=C:\Windows\System32`, `VK_DRIVER_FILES` vers l'ICD Intel et `VK_ICD_FILENAMES` vers l'ICD Intel. Toutes les variantes échouent avant adapter avec la même erreur Dawn/Vulkan.
- `webgpu@latest` sur npm est déjà `0.4.0`, identique à la version utilisée par le dépôt. Il n'existe donc pas de mise à jour npm simple à tester.
- Les artefacts GitHub Actions publics les plus récents de `dawn-gpu/node-webgpu` pointent vers le même commit `2f5fb632d9d9db9bcb3705bd868de69a6a2ae4e2` que la branche `v0.4.0`; ils ne représentent pas une build plus récente que la version publiée.
- Un build local de `node-webgpu` depuis les sources n'est pas faisable dans l'environnement courant sans préparation supplémentaire : `cmake`, `go`, `cl` et `vcvars64.bat` ne sont pas disponibles dans le shell.
- Les versions npm `webgpu@0.3.7`, `0.3.8`, `0.3.9` et `0.3.10` échouent aussi à charger Vulkan avec `Windows Error: 87`. Les versions `0.3.0`, `0.2.12` et `0.2.0` ne chargent pas correctement le module natif sous Node 20 dans ce test. Le downgrade npm n'est donc pas un contournement observé.
- Le spike Docker Desktop / WSL2 n'a pas donné d'accès Vulkan hardware exploitable. Docker Desktop a démarré avec un moteur Linux, et un conteneur peut recevoir `/dev/dxg` via `--device /dev/dxg` avec les bibliothèques WSL montées depuis `/usr/lib/wsl`. En revanche, les images Ubuntu 24.04 et Debian Trixie avec `vulkan-tools` et `mesa-vulkan-drivers` ne voient que `llvmpipe` (`PHYSICAL_DEVICE_TYPE_CPU`, `DRIVER_ID_MESA_LLVMPIPE`) dans `vulkaninfo --summary`. `/dev/dri` reste absent, y compris avec `--privileged`. Une image Rust dockerisée ne donnerait donc pas, dans cet état, un backend Vulkan matériel Intel Arc.
- Le spike Rust natif `tools/wgpu-vulkan-probe` donne un signal différent de Docker et de `node-webgpu`. Avec `wgpu@29.0.1`, le toolchain `stable-x86_64-pc-windows-gnullvm` et `rust-lld`, le process Windows voit l'Intel Arc 140V via le backend Vulkan, crée un device et écrit dans un buffer GPU. Cela confirme que Vulkan hardware est utilisable depuis un process natif non-Dawn/node-webgpu sur cette machine.
- Le même spike Rust natif passe une passe render depth-only synthétique en Vulkan, avec un process long-lived, une texture depth 4096x4096, 196424 triangles générés, le `lightMVP` solaire porté depuis le backend TypeScript, un azimut variable par itération et une attente explicite du GPU à chaque frame. Le run 200 itérations passe (`median_ms=3.32`, `max_ms=74.52`). Cela confirme que la piste Vulkan native fonctionne au-delà d'un simple test adapter/device.
- Le spike Rust natif charge aussi les caches binaires existants du VBO bâtiment via `--mesh-bin`. Un VBO réel de 716174 triangles passe 100 itérations depth-only en 4096x4096 avec `lightMVP` et azimut variable (`median_ms=5.00`, `max_ms=12.60`). Un VBO plus large de 2694178 triangles passe aussi 100 itérations (`median_ms=11.02`, `max_ms=58.16`). Ce test exerce la géométrie réelle, le vrai transform solaire côté shader et le backend Vulkan/wgpu, mais pas encore le compute shader ni le contrat complet d'une tuile.
- Le mode Rust natif `--mode=shadow` ajoute une passe compute après le rendu depth-only : lecture d'une `texture_depth_2d`, storage buffer de points, écriture d'un bitmask par `atomicOr`, copie vers un buffer de readback puis lecture CPU. Le run synthétique avec 196424 triangles, une depth texture 2048x2048, 8192 points et un azimut variable passe 10 itérations (`median_ms=2.68`, `max_ms=6.14`, dernier résultat `1720/8192` points ombrés). Ce test valide le raccord Vulkan/wgpu render + compute + readback, mais avec des points générés depuis les bounds, pas encore les vrais points de tuile.
- Le même mode Rust natif `--mode=shadow` passe sur un VBO réel de 716174 triangles avec une depth texture 4096x4096, 32186 points synthétiques et 100 itérations (`median_ms=6.77`, `max_ms=53.49`, dernier résultat `1225/32186` points ombrés). Il passe aussi sur un VBO plus large de 2694178 triangles avec le même volume de points et 100 itérations (`median_ms=8.44`, `max_ms=54.42`, dernier résultat `4065/32186` points ombrés). Ce résultat est le meilleur signal obtenu jusque-là pour une accélération GPU compute native sur la machine Windows Intel Arc, mais il ne remplace pas encore un test de contrat de tuile complet.
- Le probe Rust natif peut maintenant consommer un export de points réels via `--points-bin` et le frustum focalisé via `--focus-bounds` / `--focus-max-height`. L'exporteur `scripts/benchmark/export-wgpu-vulkan-probe-input.ts` réutilise `buildTilePoints`, `tile-grid-metadata` et l'origine du cache VBO existant pour écrire les points en `vec4<f32>` relatifs au mesh. Sur la tuile Lausanne `e2538000_n1152250_s250`, le VBO focalisé à 500 m contient 1806 obstacles et 196424 triangles, et l'export contient 32186 vrais points outdoor avec élévation. Le run Rust `--mode=shadow` en 4096x4096, avec sampling compute 2x2 aligné sur le shader TypeScript et azimut variable, passe 100 itérations (`median_ms=3.97`, `max_ms=38.89`, dernier résultat `11036/32186` points ombrés). Cela couvre maintenant la géométrie réelle, les vrais points de tuile, le frustum focalisé, le render pass, le compute shader et le readback dans un process Vulkan natif long-lived ; il manque encore l'intégration precompute complète, l'IPC éventuel et la comparaison de justesse avec le backend existant.
- Le probe Rust natif a maintenant un mode `--mode=server` avec un protocole JSON Lines sur stdin/stdout. Au démarrage, il charge le mesh, les points, la texture depth, les buffers et les pipelines une seule fois, puis accepte des commandes `evaluate` et `shutdown`. Les logs humains sont écrits sur stderr pour garder stdout exploitable par un client. Sur la même tuile Lausanne `e2538000_n1152250_s250`, le serveur passe 100 commandes `evaluate` consécutives dans un seul process (`resultCount=100`, `exitCode=0`, `median_ms=6.34`, `min_ms=3.91`, `max_ms=24.79`, dernier résultat `11036/32186` points ombrés). Cela valide le contour long-lived sans redémarrage par tuile ; il manque encore le client TypeScript de production et la comparaison de justesse.
- La comparaison de justesse `scripts/benchmark/wgpu-vulkan-vs-gpu-raster-building-benchmark.ts` isole le masque bâtiments entre le serveur Rust/wgpu Vulkan et le backend `gpu-raster` existant. Les premières divergences fortes ont permis d'identifier deux écarts de portage WebGPU : la profondeur clip WebGPU doit être en `0..1` et l'échantillonnage de la texture depth doit inverser l'axe Y par rapport au chemin WebGL/readPixels. Après correction, la passe complète du 2026-04-13 sur Lausanne `e2538000_n1152250_s250`, 06:00-21:00 Europe/Zurich au pas de 15 minutes, 53 frames au-dessus de l'horizon, 32186 vrais points DEM/outdoor, 1806 obstacles, 196424 triangles et une texture 4096x4096 donne `8/1705858` bits différents (`0.000469%`). Les temps médians observés sont `37.87 ms` pour `gpu-raster` et `3.84 ms` pour Rust/wgpu Vulkan (`speedupMedian=9.86x`, Rust `p95=7.20 ms`, `max=9.66 ms`). Ce résultat est un signal fort pour le masque bâtiments, mais il ne valide pas encore un remplacement complet du precompute : l'horizon terrain et la végétation restent volontairement hors de ce benchmark, car ils ne sont pas remplacés par ce backend.
- Le client TypeScript expérimental `scripts/benchmark/wgpu-vulkan-rust-server-client.ts` extrait maintenant la plomberie du serveur Rust : build du probe, environnement Rust local, `spawn()` long-lived, protocole JSON Lines, gestion de backpressure et shutdown. Le même benchmark peut donc lancer `--full-sunlight=true` pour garder DEM, horizon terrain adaptatif et végétation réels tout en ne remplaçant que le masque bâtiments. Sur la même tuile Lausanne `e2538000_n1152250_s250`, la passe complète 06:00-21:00 du 2026-04-13 donne `6/1705858` bits différents sur le masque bâtiments (`0.000352%`) et `3/1705858` bits différents sur le masque soleil final (`0.000176%`). La méthode terrain utilisée est `copernicus-dem30-runtime-raycast-v1|adaptive-shared-v1`, avec 2 tuiles de végétation, 32186 points outdoor, 1806 obstacles et 196424 triangles. Les temps médians observés pour la passe bâtiments sont `49.36 ms` côté `gpu-raster` et `7.57 ms` côté Rust/wgpu Vulkan (`speedupMedian=6.52x`, Rust `p95=17.93 ms`, `max=25.96 ms`). Cette passe valide beaucoup mieux le contrat precompute, mais elle reste un benchmark de comparaison : elle ne branche pas encore Rust/wgpu Vulkan dans l'écriture de cache ni dans l'orchestration multi-tuiles.
- Une mini-matrice multi-tuiles, toujours avec `--full-sunlight=true`, renforce ce signal sans encore constituer une validation de production. Sur `e2537750_n1152250_s250`, 12 frames du matin, 48131 points outdoor, 1739 obstacles et 182369 triangles donnent `4/577572` bits différents sur le masque bâtiments (`0.000693%`) et `1/577572` bit différent sur le masque soleil final (`0.000173%`), avec une médiane Rust `6.91 ms` contre `58.28 ms` côté `gpu-raster`. Sur `e2538250_n1152500_s250`, 12 frames du matin, 37734 points outdoor, 1751 obstacles et 178817 triangles donnent `3/452808` bits différents sur le masque bâtiments (`0.000663%`) et `2/452808` bits différents sur le masque soleil final (`0.000442%`), avec une médiane Rust `7.74 ms` contre `59.77 ms` côté `gpu-raster`.
- Le 2026-04-14, un premier chemin precompute explicitement flaggé a été ajouté pour Rust/wgpu Vulkan, sans toucher au script de precompute région en cours d'usage. Le mode interne `MAPPY_BUILDINGS_SHADOW_MODE=rust-wgpu-vulkan` crée un backend batch `RustWgpuVulkanShadowBackend`, lance le serveur natif par `spawn()`, garde le mesh et les points dans des fichiers runtime temporaires, supprime ces fichiers au shutdown, puis évalue les frames via JSON Lines. Le script `scripts/precompute/precompute-rust-wgpu-vulkan-dry-run.ts` force ce mode pour une seule tuile et ne persiste pas le cache soleil normal ; `--write-artifact` reste un opt-in de diagnostic.
- Le même jour, un smoke dry-run a d'abord été exécuté sur Lausanne `e2538000_n1152250_s250`, fenêtre `12:00-12:15`, avec `64/32186` points outdoor conservés en mémoire pour limiter le coût du test. Ce premier run a révélé un piège de cycle de vie : le cleanup appelé depuis une autre instance de module ne voyait pas le cache du backend. Le cleanup passe désormais par `sunlight-tile-service`, c'est-à-dire par la même instance que celle qui a construit les sources GPU.
- Après correction du cleanup, le même dry-run passe sur la tuile complète `e2538000_n1152250_s250`, toujours sans écriture de cache, avec `32186/32186` points outdoor, 1 frame, 1806 obstacles et 196424 triangles. Le run observé sort avec `FULL_TILE_DRY_RUN_EXIT=0`, méthode `dxf-footprint-prism-v5-source-order-hull-cell-maxz|rust-wgpu-vulkan-v1|tile-allowlist-v1`, `11.6s` au total, dont `1.9s` en évaluation, et aucun `mappyhour-wgpu-vulkan-probe.exe` résiduel après shutdown.
- Le dry-run accepte aussi `--mode=gpu-raster|rust-wgpu-vulkan` afin de comparer le même chemin `computeSunlightTileArtifact` sur des tuiles chaudes, sans écriture du cache soleil normal. Sur la tuile top-priority Lausanne `e2538000_n1152500_s250`, fenêtre `12:00-15:00`, 12 frames et `25656` points outdoor, le run observé donne `6.8s` en `gpu-raster` (`1.6s` d'évaluation) contre `6.2s` en Rust/wgpu Vulkan (`1.8s` d'évaluation). Cela indique que le gain du backend bâtiment ne se traduit pas automatiquement en accélération large sur une tuile chaude courte : le coût fixe de setup et les autres passes du precompute dominent encore. La comparaison doit donc être étendue à des fenêtres plus longues et à plusieurs tuiles.
- Le runner `scripts/benchmark/precompute-hot-tiles-gpu-mode-matrix.ts` formalise cette comparaison : il lit la sélection `high-value-tile-selection.top-priority.json`, lance le dry-run en `gpu-raster` puis en Rust/wgpu Vulkan, et écrit un rapport de diagnostic sans toucher au cache soleil normal. Un premier run top 3 sur `12:00-15:00` a passé toutes les tuiles sans process GPU résiduel, mais un outlier de temps Rust a été observé sur une tuile puis non reproduit en relance isolée. Cette matrice doit donc être répétée avant d'en tirer une conclusion de performance.
- Le même runner écrit maintenant, quand `--compare-values=true`, une summary compacte des valeurs par backend : compteurs de points, frames et masques base64, sans les diagnostics complets par point. Cela permet de comparer les valeurs `gpu-raster` et Rust/wgpu Vulkan sans produire deux artefacts complets très volumineux. Un smoke sur `e2538000_n1152500_s250`, fenêtre `12:00-12:15` et `64` points outdoor, donne `sunDiff=0/64` et `buildingsDiff=0/64`. Ce n'est pas une validation suffisante, mais cela confirme que le runner compare désormais la justesse en plus des temps.
- Sur les trois tuiles top-priority Lausanne, fenêtre `12:00-15:00`, vrais points outdoor et 12 frames, la comparaison de valeurs n'est pas encore nulle : `sunDiff=534/307872` (`0.173449%`) et `buildingsDiff=1034/307872` (`0.335854%`) sur `e2538000_n1152500_s250`, `sunDiff=711/344400` (`0.206446%`) et `buildingsDiff=1175/344400` (`0.341173%`) sur `e2537750_n1152500_s250`, `sunDiff=1339/372936` (`0.359043%`) et `buildingsDiff=2027/372936` (`0.543525%`) sur `e2537750_n1152750_s250`. Les masques terrain et végétation sont identiques dans ce run (`0` diff), donc l'écart vient bien du masque bâtiment. Cette divergence reste faible en pourcentage, mais elle bloque une promotion en remplacement pur tant que les points concernés n'ont pas été localisés et qualifiés.
- Le runner accepte aussi `--mode=detailed` pour comparer le chemin CPU détaillé. Ce mode utilise le vérificateur CPU sur maillages DXF/BVH après la sélection de candidats, et sert de référence de diagnostic, pas de backend de production. Sur un smoke `64` points, `12:00-12:15`, les trois modes (`gpu-raster`, Rust/wgpu Vulkan, `detailed`) donnent `0` diff. Sur la même tuile avec `2048` points et 1 frame, Vulkan reste plus proche de `gpu-raster` (`buildingsDiff=4/2048`) que du CPU détaillé (`rustVsDetailed=11/2048`, `rasterVsDetailed=13/2048`). Sur la tuile complète `e2538000_n1152500_s250`, 1 frame et `25656` points, le CPU détaillé est très lent (`176s` d'évaluation) et ne colle pas mieux à Vulkan : `rasterVsRust buildings=98/25656` (`0.381977%`), `rasterVsDetailed buildings=218/25656` (`0.849704%`), `rustVsDetailed buildings=182/25656` (`0.709386%`).
- La mini-matrice top 3 limitée à `2048` points et 1 frame donne le même signal : `rustVsDetailed` n'est pas inférieur de façon nette à `rasterVsRust`. Par exemple `e2538000_n1152500_s250` donne `rasterVsRust buildings=4/2048`, `rasterVsDetailed buildings=13/2048`, `rustVsDetailed buildings=11/2048`; `e2537750_n1152500_s250` donne `11/2048`, `15/2048`, `12/2048`; `e2537750_n1152750_s250` donne `12/2048`, `18/2048`, `16/2048`. Cela suggère que l'écart n'est pas simplement "Vulkan suit la vérité CPU et le raster actuel se trompe" ; il faut plutôt localiser les points divergents et comprendre les conventions raster/depth ou les cas de bord géométriques.
- Le rapport de comparaison contient maintenant les coordonnées minimales des points outdoor divergents (`outdoorIndex`, LV95, lat/lon, élévation), avec un top des points divergents et des exemples par masque/frame. Sur `e2538000_n1152500_s250`, 1 frame, les divergences bâtiment `gpu-raster` vs Rust/wgpu Vulkan sont bidirectionnelles (`67` points bloqués seulement par `gpu-raster`, `31` seulement par Rust/wgpu Vulkan), et les points couvrent presque toute la tuile (`E=2538000.5..2538249.5`, `N=1152506.5..1152744.5`). En revanche, les comparaisons avec le CPU détaillé sont unidirectionnelles sur ce run : `gpu-raster` bloque `218` points que `detailed` ne bloque pas, et Rust/wgpu Vulkan en bloque `182` que `detailed` ne bloque pas. Cela renforce l'idée que le CPU détaillé ne sert pas de juge simple ; il est probablement plus strict/moins couvrant dans ces cas de bord.
- Le script `scripts/benchmark/export-precompute-divergence-geojson.ts` exporte ces écarts en `FeatureCollection` GeoJSON pour inspection sur carte. Sur la même tuile/frame, les exports `divergence-gpu-raster-vs-rust-wgpu-vulkan-buildings-fulltile-1frame.geojson`, `divergence-gpu-raster-vs-detailed-buildings-fulltile-1frame.geojson` et `divergence-rust-wgpu-vulkan-vs-detailed-buildings-fulltile-1frame.geojson` contiennent respectivement `98`, `218` et `182` points, sans troncature, avec les propriétés `leftMode`, `rightMode`, `direction`, `outdoorIndex`, LV95, lat/lon et élévation. L'analyse trois-modes des mêmes summaries donne `151` points bloqués par `gpu-raster` et Rust/wgpu Vulkan mais pas par `detailed`, `67` points bloqués seulement par `gpu-raster`, `31` seulement par Rust/wgpu Vulkan, et aucun point bloqué seulement par `detailed`. C'est l'entrée de travail pour qualifier les faux positifs/faux négatifs au contact des bâtiments réels.
- Le script `scripts/benchmark/export-precompute-divergence-viewer.ts` génère aussi un viewer HTML autonome à partir des trois summaries. Sur `e2538000_n1152500_s250`, il embarque les `249` points divergents trois-modes (`R--`, `-V-`, `RV-`) et `155` bâtiments de l'index local autour de la tuile. Comme l'index actif ne stocke plus les footprints, le viewer affiche les bboxes SwissTopo, explicitement marquées `geometryKind=bbox`, afin de fournir un contexte spatial sans prétendre remplacer l'inspection du mesh DXF source.
- Le chemin d'écriture cache a aussi été durci : le worker de precompute accepte un message `shutdown`, dispose le backend via `sunlight-tile-service`, et l'orchestrateur séquentiel appelle le même cleanup en `finally`. Un smoke réel avec `MAPPY_BUILDINGS_SHADOW_MODE=rust-wgpu-vulkan`, `MAPPY_PRECOMPUTE_WORKERS=1`, Lausanne `e2538000_n1152500_s250`, `2026-04-14`, fenêtre `12:00-12:15`, a écrit une tuile puis arrêté le serveur natif (`Native server stopped`) sans process GPU résiduel. Le répertoire de cache de ce smoke a ensuite été supprimé pour ne pas laisser de run de test partiel.
- `backend=d3d11` est disponible comme contrôle : le cycle minimal Dawn/device/compute passe et le probe render-only passe 50 rendus avec le VBO réduit (~13k triangles) comme avec le VBO filtré plus large (~196k triangles). Cela renforce l'hypothèse que le crash render-only observé est lié au chemin D3D12 plutôt qu'à tout rendu WebGPU.
- Le probe `benchmark:webgpu:depth-compute` isole le shader compute, la lecture d'une `texture_depth_2d`, les storage buffers, les atomics et le readback, sans mesh bâtiment ni IPC. Il passe 50 itérations en D3D12 et D3D11 à 8192 points comme à 32186 points. Il passe aussi quand la depth texture est clearée à chaque itération. Le compute synthétique et le volume de points ne suffisent donc pas à reproduire le crash.
- Le probe `benchmark:webgpu:direct-tile-smoke` exécute la vraie tuile et le vrai VBO bâtiment dans le même process, sans worker IPC. En D3D12, la tuile complète crashe encore en `0xC0000005` vers l'itération 19/50. L'IPC n'est donc pas nécessaire pour reproduire le crash D3D12.
- `backend=d3d11` ne suffit pas encore comme contournement complet : il passe des runs directs de 20 itérations sur la tuile complète, mais un run direct plus long de 50 itérations a aussi crashé en `0xC0000005`. Il ne doit donc pas être promu sans nouvelle matrice de stabilité.
- Le client IPC avait un bug indépendant du GPU : `stdin.write()` retournant `false` était traité comme une erreur, alors que c'est un signal de backpressure Node. Ce bug est visible dès que le payload de points devient assez gros.
- Les timeouts internes de startup/init du worker IPC n'étaient pas nettoyés, ce qui gardait le process Node vivant jusqu'à 120 s après un calcul terminé.
- Le worker `webgpu-compute` initialisait son VBO avant de connaître la tuile cible ; il chargeait donc le mesh régional complet (~168k obstacles). Le filtrage spatial par focus doit être fait dès l'initialisation du worker, pas après.
- Avec filtrage spatial agressif (`MAPPY_WEBGPU_FOCUS_MARGIN_METERS=500`) et batches partiels (8k à 16k points), plusieurs évaluations répétées tiennent sans crash.
- Une tuile complète (~32k points outdoor) reste instable en worker long-lived : le process GPU finit encore par sortir avec `3221225477` (`0xC0000005`, access violation Windows) après plusieurs évaluations. Le problème n'est donc pas seulement l'IPC ni seulement la taille du VBO régional complet.
- Le probe render-only filtré réduit encore le cas minimal : le crash se reproduit sans IPC et sans compute shader. Un simple rendu depth-only répété du mesh bâtiment via Dawn/D3D12 finit par sortir avec `0xC0000005`, même avec un VBO très réduit (~13k triangles). La taille du mesh influence le nombre d'itérations avant crash, mais ne supprime pas le problème.

Conclusion provisoire : le réessai encadré reste pertinent, mais `webgpu-compute` ne peut pas être promu tant qu'on n'a pas isolé ou contourné l'instabilité du render pass depth-only Dawn/D3D12 sur le vrai mesh bâtiment. Une stratégie de chunking compute, de correction IPC ou de passage opportuniste en D3D11 ne suffit pas. La piste la plus prometteuse observée depuis le 2026-04-13 est Rust/wgpu Vulkan natif. Le chemin precompute flaggé existe maintenant, passe une tuile complète en dry-run et sait écrire une tuile de smoke avec cleanup explicite, mais il reste expérimental. Le CPU détaillé ne valide pas pour l'instant Vulkan contre `gpu-raster` : il diffère lui aussi et coûte beaucoup trop cher pour servir de remplacement. Les écarts de masque bâtiment sont désormais localisables par point ; le prochain verrou est de les visualiser/rejouer sur quelques cas géométriques, puis de relancer une matrice multi-tuiles/multi-jours avant toute écriture de cache de production.

## Conséquences

Positives :

- Le pipeline de production reste stable et s'appuie sur le chemin déjà documenté.
- Le réessai GPU cible les causes probables : cycle de vie Dawn, backend D3D12/Vulkan, isolation de process et I/O.
- On évite la boucle coûteuse "crash puis restart" qui rendait chaque tuile trop chère.

Tradeoffs :

- `webgpu-compute` n'est pas promu en production tant qu'une matrice de reproduction n'est pas verte.
- Le sous-processus long-lived ajoute un protocole IPC et une discipline stricte sur les données transférées.
- Les sources externes ne suffisent pas à attribuer le crash à Intel Arc de manière définitive.

## Plan de vérification avant promotion

Avant de remettre `webgpu-compute` dans un chemin supporté, exécuter au minimum :

1. Test minimal de cycle de vie Dawn : créer adapter/device, lancer un render ou compute trivial, appeler `destroy()`, nettoyer les références, sortir proprement.
2. Smoke test tuile : calculer une tuile Lausanne sans écriture de cache depuis le process GPU.
3. Test de stabilité batch : répéter beaucoup d'évaluations tuile/frame dans le même process GPU long-lived.
4. Comparaison backend : exécuter la même matrice en `d3d12` et en `vulkan` sur la même machine et le même driver ; utiliser `d3d11` seulement comme contrôle si Vulkan ne charge toujours pas.
5. Comportement de panne : vérifier timeout, `device.lost` quand disponible, et fallback vers `gpu-raster`/CPU sans boucle de redémarrage par tuile.

## Hors scope

- Changer le backend de production.
- Ajouter une nouvelle API publique.
- Promouvoir `MAPPY_WEBGPU_BACKEND` ou `MAPPY_WEBGPU_DLLDIR` en API publique.
- Conclure que notre crash est un bug Intel Arc publiquement prouvé.
